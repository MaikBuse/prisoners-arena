# v1.7 — Commit-Reveal Strategies

Status: 🟢 Complete

**Motivation:** Strategies are stored on-chain in plaintext at registration time. Any participant (or observer) can read all entered strategies before the tournament runs, enabling counter-picking — choosing a strategy specifically to exploit what others have submitted. This undermines the game-theoretic integrity of the tournament. A commit-reveal scheme hides strategies until registration closes, forcing players to choose blind, which is the intended design of Prisoner's Dilemma tournaments.

**Depends on:** v1.4 (Strategy Parameters) deployed. v1.5 and v1.6 are independent.

---

## Design Overview

### New Tournament Flow

```
Registration → Reveal → Running → Payout
```

| Phase | What Happens | Duration |
|-------|-------------|----------|
| **Registration** | Players submit hash commitment + stake. Strategy hidden. | Config-defined (existing) |
| **Reveal** | Players submit plaintext strategy + params + salt. Contract verifies hash. | Config-defined (new, e.g., 48h) |
| **Running** | Operator cranks matches as before. All strategies now public. | Same as current |
| **Payout** | Winners claim prizes. Same as current. | Same as current |

### Commitment

```
commitment = SHA256(strategy_byte || param_bytes[5] || salt[16])
```

- `strategy_byte`: 1 byte (Strategy enum index)
- `param_bytes`: 5 bytes (StrategyParams, in Borsh order: forgiveness, retaliation_delay, noise_tolerance, initial_moves, cooperate_bias)
- `salt`: 16 bytes, player-chosen random nonce

Total preimage: 22 bytes. SHA256 output: 32 bytes.

**Why 16-byte salt:** The strategy+params space is small (~9 × 100^5 ≈ 90 billion combinations). Without a salt, an attacker can precompute all hashes and reverse any commitment. A 16-byte random salt makes brute-force infeasible (2^128 possibilities per commitment).

## 6. New `reveal_strategy` Instruction (`instructions/player.rs`)

### Instruction

```rust
pub fn reveal_strategy(
    ctx: Context<RevealStrategy>,
    strategy: Strategy,
    params: StrategyParams,
    salt: [u8; 16],
) -> Result<()> {
    let tournament = &mut ctx.accounts.tournament;
    let entry = &mut ctx.accounts.entry;
    let clock = Clock::get()?;

    // State check
    require!(
        tournament.state == TournamentState::Reveal,
        DilemmaError::InvalidState
    );

    // Deadline check
    require!(
        clock.unix_timestamp <= tournament.reveal_ends,
        DilemmaError::RevealPeriodEnded
    );

    // Not already revealed
    require!(!entry.revealed, DilemmaError::AlreadyRevealed);

    // Validate params (same as v1.4 enter_tournament validation)
    require!(params.forgiveness <= 100, DilemmaError::InvalidParams);
    require!(params.retaliation_delay <= 10, DilemmaError::InvalidParams);
    require!(params.noise_tolerance <= 5, DilemmaError::InvalidParams);
    require!(params.cooperate_bias <= 100, DilemmaError::InvalidParams);

    // Verify commitment
    let mut preimage = Vec::with_capacity(22);
    preimage.push(strategy as u8);
    preimage.push(params.forgiveness);
    preimage.push(params.retaliation_delay);
    preimage.push(params.noise_tolerance);
    preimage.push(params.initial_moves);
    preimage.push(params.cooperate_bias);
    preimage.extend_from_slice(&salt);

    let hash = solana_program::hash::hash(&preimage);
    require!(
        hash.to_bytes() == entry.commitment,
        DilemmaError::CommitmentMismatch
    );

    // Store revealed strategy
    entry.strategy = strategy;
    entry.strategy_params = params;
    entry.revealed = true;

    // Update tournament vecs
    let idx = entry.index as usize;
    tournament.strategies[idx] = strategy as u8;
    tournament.strategy_params[idx] = params;

    // Track progress
    tournament.reveals_completed += 1;

    Ok(())
}
```

### Accounts

```rust
#[derive(Accounts)]
pub struct RevealStrategy<'info> {
    #[account(mut, has_one = tournament)]
    pub entry: Account<'info, Entry>,
    #[account(mut)]
    pub tournament: Account<'info, Tournament>,
    #[account(mut, constraint = player.key() == entry.player @ DilemmaError::Unauthorized)]
    pub player: Signer<'info>,
}
```

No SOL transfer, no realloc. Just data mutation.

### Compute

`solana_program::hash::hash` (SHA256) costs ~100 CUs for 22 bytes. Total instruction ~5,000 CUs. Trivial.

## 9. New `forfeit_unrevealed` Instruction (`instructions/tournament.rs`)

Called by operator for each unrevealed entry after reveal deadline. Processes forfeiture: stake goes to prize pool, entry is marked.

This must be a **per-entry instruction** (not batch) because each entry PDA needs to be passed as an account for rent recovery.

```rust
pub fn forfeit_unrevealed(ctx: Context<ForfeitUnrevealed>) -> Result<()> {
    let tournament = &mut ctx.accounts.tournament;
    let entry = &mut ctx.accounts.entry;
    let clock = Clock::get()?;

    require!(
        tournament.state == TournamentState::Reveal,
        DilemmaError::InvalidState
    );

    require!(
        clock.unix_timestamp > tournament.reveal_ends,
        DilemmaError::RevealPeriodNotEnded
    );

    require!(!entry.revealed, DilemmaError::AlreadyRevealed);

    // Forfeit: stake stays in tournament pool (benefits remaining players)
    // Mark entry
    let idx = entry.index as usize;
    tournament.strategies[idx] = u8::MAX;  // sentinel stays
    tournament.players[idx] = Pubkey::default(); // mark as removed

    // Decrement active count, increment forfeit count
    tournament.participant_count -= 1;
    tournament.forfeits += 1;

    // Close entry PDA — rent goes to operator (compensates tx cost)
    // Entry account is closed via close = operator constraint in Accounts struct

    Ok(())
}
```

### Accounts

```rust
#[derive(Accounts)]
pub struct ForfeitUnrevealed<'info> {
    #[account(
        mut,
        has_one = tournament,
        close = operator,  // rent → operator
    )]
    pub entry: Account<'info, Entry>,
    #[account(mut)]
    pub tournament: Account<'info, Tournament>,
    #[account(
        mut,
        constraint = operator.key() == config.operator @ DilemmaError::Unauthorized
    )]
    pub operator: Signer<'info>,
    pub config: Account<'info, Config>,
}
```

### Stake Handling

The forfeited player's stake is **already in the tournament pool** (transferred at `enter_tournament`). By not refunding it and decrementing `participant_count`, the per-winner share increases. This is the penalty: your stake subsidizes the winners.

## 10. Update `claim_refund` (`instructions/player.rs`)

Refunds are still available during Registration (unchanged) **and now also during Reveal** — a player who committed can still back out before revealing. They get their stake back but their strategy remains hidden (they never revealed it).

```rust
pub fn claim_refund(ctx: Context<ClaimRefund>) -> Result<()> {
    let tournament = &mut ctx.accounts.tournament;

    require!(
        tournament.state == TournamentState::Registration
            || tournament.state == TournamentState::Reveal,  // NEW
        DilemmaError::InvalidState
    );

    // ... rest unchanged ...
}
```

**Design decision:** Allowing refund during Reveal is player-friendly. A player who realizes they lost their salt can bail out instead of being forfeited. The tradeoff: someone could commit, see who else entered (pubkeys are public), then refund if they don't like the field. This is acceptable — they can't see *strategies*, only *who* entered, which was already visible during Registration.

### New Lifecycle Steps

The operator automates the tournament lifecycle. New steps:


### `forfeit_unrevealed` Loop

After reveal deadline, operator scans all entries for the tournament:

```rust

    if !entry.revealed {
        send_tx(forfeit_unrevealed(entry));
    }
}
```

Then calls `close_reveal`.

## Risk Assessment

- **Game-theoretic improvement:** Eliminates counter-picking. Strategies are chosen blind, as intended by IPD tournament design. Significant integrity upgrade.
- **UX cost:** Two transactions instead of one. Players must return during the reveal window. Mitigated by generous reveal duration (48h default) and frontend salt backup in localStorage.
- **Salt loss risk:** If a player loses their salt, they cannot reveal and forfeit their stake. Mitigations: CLI saves to file, frontend saves to localStorage, UI prominently warns "save your salt." Ultimately this is the player's responsibility — same as losing a private key.
- **Operator complexity:** Two new lifecycle steps (forfeit loop + close_reveal). Straightforward to automate but more moving parts.
- **Compute:** SHA256 verification is ~100 CUs. No compute concerns.
- **Size:** Entry grows by 33 bytes (commitment + revealed). Tournament grows by 24 bytes base. Modest.
- **Breaking change:** Yes — struct layout changes across Entry, Tournament, Config, TournamentState. Full redeploy. No migration.
- **Zero-player edge case:** If all players forfeit or refund during Reveal, `close_reveal` sees 0 active players. Should transition to Payout with empty winner set (tournament effectively cancelled, forfeited stakes go to fees via normal expiry flow). Test this explicitly.
