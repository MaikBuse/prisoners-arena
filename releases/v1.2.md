## 1. Harden `close_tournament` — Entry Validation & Unclaimed Reward Sweep (Critical)

**Problem A — No entry validation:** `close_tournament` checks state == Payout and 30 days elapsed, but never verifies all Entry accounts are closed. The `EntriesRemaining` error (6019) is defined but unused. If entries still exist when the tournament account is closed, those entries become orphaned — their `tournament` field points to a deleted account, potentially locking funds.

**Problem B — Orphaned funds:** When winners don't claim their payout within the 30-day expiry window, the operator closes their entries via `close_expired_entry`, but the unclaimed reward lamports remain in the tournament PDA. When `close_tournament` closes the account, these funds are silently sent to the operator (via `close = operator`) with no accounting trail — bypassing `accumulated_fees` entirely.

**Fix:** Two changes to `close_tournament`:

1. **Entry validation (on-chain enforcement):** Add `entries_remaining: u32` to Tournament. Increment on `enter_tournament`, decrement on `claim_payout`, `claim_refund`, and `close_expired_entry`. `close_tournament` requires `entries_remaining == 0`.

2. **Unclaimed reward sweep:** After entry validation passes, sweep any remaining lamports above rent-exempt minimum from the tournament PDA to the config PDA and increment `accumulated_fees`:

```rust
let surplus = tournament_info.lamports()
    .checked_sub(min_balance)
    .unwrap_or(0);
if surplus > 0 {
    **tournament_info.try_borrow_mut_lamports()? -= surplus;
    **config_info.try_borrow_mut_lamports()? += surplus;
    config.accumulated_fees = config.accumulated_fees
        .checked_add(surplus)
        .ok_or(DilemmaError::Overflow)?;
}
```

**Tradeoff:** The sweep is a lump sum — can't distinguish unclaimed rewards from rounding dust. Acceptable for v1 where the operator is trusted.

**Tests required:**

- Entry counter increments/decrements correctly across `enter_tournament`, `claim_payout`, `claim_refund`, `close_expired_entry`
- `close_tournament` rejects when `entries_remaining > 0`
- `close_tournament` sweeps unclaimed reward lamports to `accumulated_fees`
- Verify swept amount matches expected unclaimed rewards (full lifecycle: enter → finalize → expire some → close)
- `withdraw_fees` succeeds after sweep (end-to-end flow)

**Scope:** Contract change — new Tournament field, validation in `close_tournament`, config PDA as additional account for the sweep.

## 2. Fix SlotHash Randomness Extraction

**Problem:** `close_registration` reads the first 32 bytes of the SlotHashes sysvar:

```rust
seed.copy_from_slice(&slot_hashes_data[..32]);
```

The SlotHashes sysvar layout starts with a `u64` length prefix (8 bytes), then `(Slot, Hash)` pairs (8 + 32 bytes each). The current code reads the length field + partial first slot number, not an actual hash. Randomness quality is lower than intended — partly deterministic.

**Fix:** Skip the 8-byte length prefix and read the first hash (bytes 16..48):

```rust
// Skip: 8 bytes (vec len) + 8 bytes (first slot number) = 16
require!(slot_hashes_data.len() >= 48, DilemmaError::SlotHashUnavailable);
seed.copy_from_slice(&slot_hashes_data[16..48]);
```

**Impact:** Correctness. Not exploitable in v1 (operator is trusted), but wrong is wrong.

**Scope:** Contract change (one-line fix).

## 3. Add Multi-Batch `run_matches` Integration Test

**Problem:** The v1.1 batch index double-counting bug was invisible in single-batch tests. The fix (snapshotting `start_index`) is deployed, but no regression test exists. This is the most dangerous class of bug — it only manifests at scale.

**Fix:** Add a contract-level integration test that:

1. Sets up a tournament with enough matches to require 2+ batches
2. Runs `run_matches` (batch 1: indices 0-4)
3. Verifies `matches_completed == 5`
4. Runs `run_matches` (batch 2: indices 5-9)
5. Verifies all 10 unique match indices executed with correct scores
6. Confirms no index skipping or double-counting

**Scope:** Test only. No contract change.

## 5. Add API Rate Limiting

**Problem:** No rate limiting on the frontend API. PLAN.md specifies 60 req/min per IP but it's unimplemented. The API could be trivially DoS'd, which matters when it's serving live tournament data backed by real SOL.

**Fix:** Add rate limiting middleware to Next.js API routes. Options:

- **A) In-memory rate limiter** — Simple Map-based token bucket. Good enough for single-instance deployment.
- **B) Edge middleware** — If deploying to Vercel/Cloudflare, use their built-in rate limiting.

**Recommendation:** Option A for now. 60 req/min per IP, 429 response with `Retry-After` header.

**Scope:** Frontend only. No contract change.

## 7. Document Entry Rent Destination

**Problem:** `close_expired_entry` uses `close = operator`, sending entry account rent to the operator. All other fee flows go to admin. This is a minor economic inconsistency — likely intentional (compensates operator for tx costs) but undocumented.

**Fix:** Document this explicitly in the architecture spec under "Rent Handling" section. If unintentional, change `close = operator` to `close = admin` (or add rent to `accumulated_fees`).

**Scope:** Documentation (or minor contract change if behavior should change).
