# v1.4 — Strategy Parameters

Status: 🟡 In Progress

**Motivation:** The match-logic crate already supports strategy parameters (forgiveness, retaliation_delay, noise_tolerance, initial_moves, cooperate_bias) but they're never exposed to players. All entries use default params (zeros). Exposing parameters transforms the game from "pick 1 of 9 strategies" into a richer design space where players can tune behavior — creating variants like Generous TFT, Forgiving Grim Trigger, or biased Random. This significantly increases strategic depth and replayability.

---

## Current State

**On-chain (`state.rs`):**
- `Strategy` enum: 9 variants, 1 byte, no params
- `Entry.strategy`: single enum value
- `Tournament.strategies`: `Vec<u8>` storing base enum per player
- `enter_tournament` instruction: takes `strategy: Strategy` (1 byte)
- Conversion: `Strategy::into()` calls `match_logic::Strategy::new(base)` → default params (all zeros)

**Match-logic (`strategy.rs`):**
- `Strategy` = `StrategyBase` + `StrategyParams`
- `StrategyParams`: 5 fields, all `u8`
- `execute_strategy()` already uses all params in game logic
- Params are functional but unreachable from on-chain input

---

## Parameter Definitions

| Parameter | Type | Range | Default | Affects Strategies | Effect |
|-----------|------|-------|---------|-------------------|--------|
| `forgiveness` | u8 | 0–100 | 0 | TitForTat, SuspiciousTitForTat | % chance to cooperate instead of retaliating after opponent defects |
| `retaliation_delay` | u8 | 0–10 | 0 | *(reserved, not yet used in execute_strategy)* | Rounds to wait before retaliating |
| `noise_tolerance` | u8 | 0–5 | 0 | GrimTrigger | Number of opponent defections to ignore before triggering |
| `cooperate_bias` | u8 | 0–100 | 50 | Random | % chance to cooperate each round (0=always defect, 100=always cooperate). Default 50. |
| `initial_moves` | u8 | 0–255 | 0 | All | Bitmask overriding first 8 rounds (bit=1 forces Defect for that round) |

**Note on `retaliation_delay`:** Currently defined in `StrategyParams` but not referenced in any `execute_strategy` branch. Either implement it in the match-logic crate as part of this release, or document it as reserved. See item 2.

**`cooperate_bias` default fix:** The match-logic code currently treats 0 as 50 (`if params.cooperate_bias == 0 { 50 }`). This implicit magic value is removed in this release. Instead, `StrategyParams::default()` sets `cooperate_bias: 50` explicitly, and `execute_random` uses the value literally. This means 0 = always defect, 100 = always cooperate, 50 = fair coin. Players get what they ask for.

## 2. Fix `cooperate_bias` Default (Match Logic — `strategy.rs`)

Remove the implicit `0 → 50` conversion in `execute_random`:

```rust
// Current:
let bias = if params.cooperate_bias == 0 { 50 } else { params.cooperate_bias };

// New:
let bias = params.cooperate_bias;
```

Update `StrategyParams::default()`:

```rust
impl Default for StrategyParams {
    fn default() -> Self {
        Self {
            forgiveness: 0,
            retaliation_delay: 0,
            noise_tolerance: 0,
            initial_moves: 0,
            cooperate_bias: 50, // was 0 with implicit conversion
        }
    }
}
```

This is a **behavior-preserving change** — `Strategy::new(base)` calls `StrategyParams::default()` which now explicitly sets 50, producing the same runtime behavior as the old `0 → 50` hack. But `cooperate_bias=0` sent by a player now means 0% cooperation, not 50%.

## 3. Implement or Remove `retaliation_delay` (Match Logic — `strategy.rs`)

`retaliation_delay` is defined in `StrategyParams` but unused in `execute_strategy`. Two options:

**Option A — Implement it:** Add delay logic to TFT, SuspiciousTFT, and GrimTrigger. A delay of N means the strategy waits N rounds after seeing a defection before retaliating. This creates "patient" variants.

```rust
// In execute_tit_for_tat, after detecting opponent defection:
let rounds_since_defection = opponent_history.len() - last_defection_index;
if rounds_since_defection <= params.retaliation_delay as usize {
    return Move::Cooperate; // Still waiting
}
Move::Defect
```

**Option B — Remove it:** Drop from `StrategyParams`, save 1 byte. Simpler.

**Recommendation:** Option A. The parameter is already allocated and the implementation is straightforward. "Patient TFT" is a legitimate variant in the literature.

// Replace the existing From impl with a function that takes both
pub fn to_match_strategy(strategy: Strategy, params: &StrategyParams) -> match_logic::Strategy {
    let base = match strategy {
        Strategy::TitForTat => match_logic::StrategyBase::TitForTat,
        Strategy::AlwaysDefect => match_logic::StrategyBase::AlwaysDefect,
        Strategy::AlwaysCooperate => match_logic::StrategyBase::AlwaysCooperate,

        Strategy::Pavlov => match_logic::StrategyBase::Pavlov,

        noise_tolerance: params.noise_tolerance,

## Risk Assessment

- **Low risk:** Additive change. Default params (all zeros) produce identical behavior to v1.3 — backward compatible game logic.
- **Compute impact:** Negligible. Param validation is 4 extra comparisons. Strategy execution already reads params — no new compute in `run_matches`.
- **Size impact:** +5 bytes per Entry, +5 bytes per player in Tournament. At 5,000 players: +25KB. Well within limits.
- **Breaking change:** Yes — struct layout change requires redeploy. No migration possible; existing tournaments incompatible.
- **Game balance:** Parameterized strategies create a larger strategy space. `initial_moves` bitmask is the most powerful param — it allows hardcoded opening sequences that can exploit specific opponent strategies. This is intentional but worth monitoring in playtests.
- **`cooperate_bias=0` resolved:** The `if bias == 0 { 50 }` hack in `execute_random` is removed. `StrategyParams::default()` changes to `cooperate_bias: 50`. Value 0 means literally 0% cooperation. `Random(0)` ≡ AlwaysDefect, `Random(100)` ≡ AlwaysCooperate — mathematically correct and intuitive.
