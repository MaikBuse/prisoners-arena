## 1. Fix `accumulated_fees` Lamport Transfer (Critical)

**Problem:** `finalize_tournament` increments `config.accumulated_fees` but never transfers lamports from the tournament account to the config account. The counter says fees exist, but the SOL is still in the tournament PDA. `withdraw_fees` pulls from config's lamport balance, which only has rent-exempt minimum — will fail at scale.

**Fix:** In `finalize_tournament`, after calculating `house_fee + dust`, transfer that amount from the tournament account to the config account:

```rust
let fee_total = house_fee + dust;
config.accumulated_fees += fee_total;

// Actually move the lamports
**tournament.to_account_info().try_borrow_mut_lamports()? -= fee_total;
**config.to_account_info().try_borrow_mut_lamports()? += fee_total;
```

**Same issue in `close_expired_entry`:** When unclaimed winner shares are added to `accumulated_fees`, the lamports are in the tournament account, not config. Need to transfer there too:

```rust
if !entry.paid_out && entry.score >= tournament.min_winning_score {
    let unclaimed_share = tournament.winner_pool / tournament.winner_count as u64;
    config.accumulated_fees += unclaimed_share;

    // Move lamports from tournament to config
    **tournament.to_account_info().try_borrow_mut_lamports()? -= unclaimed_share;
    **config.to_account_info().try_borrow_mut_lamports()? += unclaimed_share;
}
```

**Impact:** Blocks mainnet launch. Without this, house fees are unwithdrawable.

**Scope:** Contract change (redeploy required).

## 2. Operator Retry-on-Conflict

**Problem:** Devnet RPC returns stale account data for ~5-10s after tx confirmation. Operator reads old state, sends the wrong instruction, gets `InvalidState` (error 6000).

**Fix:** In the operator's main cycle loop, catch `InvalidState` errors. On conflict: wait 3s, re-fetch state, retry the cycle. Applies to both auto and manual mode.

**Impact:** Operational reliability. No contract change.

**Scope:** Operator bot only.

## 3. K Matches for All Player Counts

**Problem:** With 2 players and K=15, only 1 match runs. With 4 players and K=15, only 6 matches run (each pair once). The `matches_per_player` parameter is silently ignored for small tournaments.

**Root cause:** `generate_all_pairings` branches to `generate_round_robin` when `n <= k + 1`. Round-robin produces `n(n-1)/2` *unique* pairings and stops — it never repeats pairs. For N=2: 1 match. For N=4: 6 matches. The K parameter is effectively capped at `n-1`.

This matters because each match already gets a **unique game seed** via `SeededRng::new(seed, match_index)`, producing different round counts and different probabilistic strategy outcomes (Random, Suspicious Tit-for-Tat thresholds, etc.). Repeated pairings are *not* duplicate games — they're meaningfully distinct encounters that let iterated strategies (Tit-for-Tat, Gradual, Pavlov) build independent histories.

**Design:** The circular offset method (large-N path) already guarantees `n×K/2` total matches. The fix targets only the small-N path where round-robin under-produces.

When `n <= k + 1` (round-robin territory), cycle through complete round-robins until every player has ≥ K matches:

1. Generate one full round-robin: all `n(n-1)/2` unique pairs. Each player gets `n-1` matches.
2. If `n-1 < K`, repeat the full round-robin. Each repetition adds `n-1` matches per player.
3. Number of full cycles needed: `ceil(K / (n-1))`.
4. Total matches: `cycles × n(n-1)/2`. Each player gets `cycles × (n-1)` matches (≥ K, may overshoot slightly for non-divisible K — acceptable, same as the odd-K rounding in the large-N path).
5. Shuffle the final match list. Each match gets a distinct `match_index` → distinct game seed.

**Examples with K=15:**
| N | Unique pairs | Cycles needed | Total matches | Matches/player |
|---|-------------|---------------|---------------|----------------|
| 2 | 1 | 15 | 15 | 15 |
| 3 | 3 | 5 | 15 | 10 (wait — see note) |
| 4 | 6 | 4 (ceil(15/3)) | 24 | 12 (wait — 4×3=12 < 15) |

Correction — N=4, K=15: `n-1 = 3`, `ceil(15/3) = 5`, total = `5 × 6 = 30`, matches/player = `5 × 3 = 15`. ✓

| N | Unique pairs | Cycles needed (ceil(K/(n-1))) | Total matches | Matches/player |
|---|-------------|-------------------------------|---------------|----------------|
| 2 | 1 | 15 | 15 | 15 |
| 3 | 3 | 8 (ceil(15/2)) | 24 | 16 |
| 4 | 6 | 5 (ceil(15/3)) | 30 | 15 |
| 5 | 10 | 4 (ceil(15/4)) | 40 | 16 |
| 6 | 15 | 3 (ceil(15/5)) | 45 | 15 |
| 16 | 120 | 1 | 120 | 15 |

For N=3 and N=5, players get 16 instead of 15 — one extra match. Same trade-off as the existing odd-K rounding in the large-N path (`offsets_needed = (k + 1) / 2` can overshoot by 1). Acceptable.

**Changes:**

`crates/match-logic/src/pairing.rs`:
- **`generate_round_robin(n, k, seed)`** — add `k` parameter. Compute `cycles = ceil(K / (n-1))`. Repeat the base round-robin `cycles` times. Return all `cycles × n(n-1)/2` pairs, shuffled.
- **`generate_all_pairings`** — pass `k` to `generate_round_robin`.
- No changes to `get_pairing_for_match` or `calculate_match_count` — they already delegate to `generate_all_pairings` and index/count the result.

`crates/match-logic/src/pairing.rs` (tests):
- Fix `test_two_players`: expect 15 matches (K=5 → `ceil(5/1) = 5` cycles × 1 pair = 5 matches... wait, test uses K=5, so 5 matches, not 1).
- Fix `test_small_tournament_round_robin`: N=4 K=5 → `ceil(5/3) = 2` cycles × 6 pairs = 12 matches, each player gets 6 matches.
- Fix `test_match_count_round_robin`: adjust expected count.
- Add `test_two_players_k15`: N=2, K=15 → 15 matches.
- Add `test_repeated_pairings_distinct_indices`: verify that repeated pairs have different match indices (ensuring different game seeds).
- Add `test_small_n_each_player_has_geq_k_matches`: parametric check across N=2..6 with K=15.

**No changes to:**
- `game.rs` / `run_match` — already uses `match_index` for unique seeding.
- Contract instructions — `close_registration` calls `calculate_match_count` which delegates to `generate_all_pairings`. The corrected count propagates automatically.
- `get_pairing_for_match` — indexes into the generated vec. Repeated pairs at different indices are returned correctly.

**Impact:** Gameplay correctness for small tournaments. Contract redeploy required (match-logic crate compiled into program).

**Scope:** match-logic crate only (pairing.rs + tests). Contract recompile, no instruction changes.

## 4. Snapshot `registration_duration`

**Problem:** `registration_duration` is read from live config, not snapshotted to the tournament. All other gameplay parameters (`stake`, `house_fee_bps`, `matches_per_player`) are snapshotted at creation. If admin changes `registration_duration` mid-tournament, deadline extensions use the new value — breaking the snapshot principle.

**Options:**
- **A) Snapshot it** — Add `registration_duration: i64` to Tournament struct. Extensions use the snapshotted value. Consistent with other fields.
- **B) Accept the inconsistency** — Document that extensions use live config intentionally (admin can adjust extension length). Simpler, no struct change.

**Recommendation:** Option A. It's a small struct addition and keeps the invariant clean. The admin can always update config for the *next* tournament.

**Impact:** Minor. Design consistency.

**Scope:** Contract change (new field in Tournament struct).

## 5. `close_tournament` Instruction

**Problem:** After payout + claim expiry, tournament accounts retain rent-exempt lamports (~0.002-0.003 SOL each) permanently. With dynamic sizing, large tournaments hold more. No way to reclaim.

**Fix:** New `close_tournament` instruction:
- **Precondition:** Tournament in Payout state, 30 days past `payout_started_at`, all entries already closed
- **Authorization:** Admin or operator
- **Action:** Zero account data, close the PDA, send all remaining lamports (base rent + any leftover) to admin
- **No house fee pool involvement** — the tournament account itself holds the remaining lamports; just drain it directly
- Operator executes this automatically for expired tournaments

**Note:** The rent deltas that players paid via realloc are already in the tournament account. They don't need to be "refunded from" anywhere — they're recovered naturally when the account is closed.

**Recovery:** ~0.002-0.003 SOL per tournament, vs ~0.000005 SOL tx cost. ~400-600× ROI.

**Impact:** Operational cost recovery. Prevents unbounded rent accumulation.

**Scope:** New contract instruction + operator automation.

## 7. Fix `run_matches` Batch Index Double-Counting (Critical)

**Problem:** `run_matches` computes `match_index = tournament.matches_completed + batch_idx` inside a loop that also increments `matches_completed`. The mutating base causes match indices to skip: batch starting at offset 10 runs indices 10, 12, 14, 16(OOB) instead of 10, 11, 12, 13, 14.

**Root cause:** The loop variable `tournament.matches_completed` is incremented each iteration, but also used as the base for the next iteration's index calculation.

```rust
// BEFORE (buggy):
for batch_idx in 0..matches_to_run {
    let match_index = tournament.matches_completed + batch_idx; // base shifts!
    ...
    tournament.matches_completed += 1; // mutates the base
}

// AFTER (fixed):
let start_index = tournament.matches_completed; // snapshot before loop
for batch_idx in 0..matches_to_run {
    let match_index = start_index + batch_idx; // stable base
    ...
    tournament.matches_completed += 1;
}
```

**Why existing tests missed it:** All contract tests ran matches from `matches_completed = 0` in a single batch. The double-counting only manifests when the starting offset is non-zero (batch 2+). A multi-batch integration test would have caught it immediately.

**Test gap:** Need a contract-level test that:
1. Runs `run_matches` with 5 matches (batch 1: indices 0-4)
2. Verifies `matches_completed = 5`
3. Runs `run_matches` again (batch 2: should be indices 5-9, not 5,7,9,11,13)
4. Verifies all 10 unique match indices were executed with correct scores

**Impact:** Critical — caused tournaments with >5 matches to skip indices and eventually fail with `InvalidMatch`. Blocks any tournament with more than one batch of matches.

**Scope:** One-line fix in contract (`tournament.rs`).
