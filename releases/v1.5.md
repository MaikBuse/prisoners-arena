# v1.5 — Matchmaking Overhaul

Status: ⚪ Planned

**Motivation:** The current matchmaking (K=15, offset-based pairing, 5–15 rounds per match) produces statistically weak results. Players face only ~16% of the field, the offset pairing method creates structural correlation between adjacent indices, and matches are too short for complex strategies (Gradual, Pavlov) to fully express themselves. This release redesigns matchmaking to approach Axelrod/Stewart-Plotkin tournament quality while respecting on-chain compute and cost constraints at scale (up to 5,000 players).

---

## Analysis: Current vs Gold Standard

| Metric | Axelrod (1984) | Stewart & Plotkin (2012) | Current (v1.3) |
|--------|---------------|------------------------|----------------|
| Pairings | Full round-robin | Full round-robin | ~16% sample (offset) |
| Rounds/match | 151–200 | 150 | 5–15 (~10 avg) |
| Total rounds/player | 2,800–9,000+ | ~1,000+ | ~160 |
| Opponent coverage | 100% | 100% | 16% |

**Key problems:**
1. **Low opponent diversity** — At K=15 with n=100, each player faces 16 of 99 opponents. Score variance from opponent sampling can exceed strategic signal, turning results into a lottery.
2. **Structural pairing bias** — The circular offset method means players with adjacent indices face structurally similar opponent neighborhoods. If strategies cluster by registration order, this amplifies bias.
3. **Short matches** — 5–15 rounds is insufficient for strategies like Gradual (needs ~10+ rounds for escalation) and Pavlov (needs multiple win-stay/lose-switch cycles to stabilize). Axelrod used 200 rounds per match for good reason.

---

## Design: Tiered Matchmaking

Matchmaking adapts based on participant count at `close_registration` time. Three tiers, each optimized for its scale.

### Tier Definitions

| Tier | Player Count | K (matches/player) | Round Range | Expected Rounds | Pairing Method |
|------|-------------|-------------------|-------------|-----------------|----------------|
| **A** | n ≤ 200 | n − 1 (full RR) | 20–50 | ~35 | All pairs |
| **B** | 201–1,000 | 99 | 20–50 | ~35 | Random sampling |
| **C** | 1,001–5,000 | 99 | 10–30 | ~20 | Offset (existing) |

### Why These Numbers

**K = n−1 for Tier A:** Full round-robin. Every player faces every other player. Eliminates all sampling variance — results are pure strategic signal. Matches Axelrod exactly. At n=200, total matches = 19,900, which costs ~$4 in operator tx fees against a pool worth ~$1,500. Trivial.

**K = 99 for Tiers B & C:** Sampling 99 opponents from a field of 9 strategy types provides excellent statistical coverage. Even if a strategy represents only 5% of the field, you'd expect ~5 opponents of that type — enough to average out per-match variance. Score standard error at K=99 is ~10% of per-match std dev. Diminishing returns beyond K=99 are minimal for ranking purposes.

**Round ranges scaled by tier:**
- Tiers A & B (20–50, ~35 avg): Gives strategies room to fully express. Gradual can escalate through 5+ retaliation cycles. Pavlov stabilizes. TFT patterns emerge clearly. Total rounds/player: up to ~7,000 (Tier A) or ~3,465 (Tier B) — comparable to Axelrod's ~2,800–9,000.
- Tier C (10–30, ~20 avg): Slight compression for cost control at scale. Total rounds/player: ~1,980 — still above Axelrod's minimum and well above the current ~160.

### Cost Model

| Scenario | Players | Total Matches | Operator Txs | Est. Cost ($) | Pool ($, @$150/SOL, 0.05 SOL stake) | Cost % |
|----------|---------|--------------|-------------|--------------|--------------------------------------|--------|
| Tier A min | 10 | 45 | 9 | $0.01 | $75 | 0.01% |
| Tier A mid | 100 | 4,950 | 990 | $1 | $750 | 0.1% |
| Tier A max | 200 | 19,900 | 3,980 | $4 | $1,500 | 0.3% |
| Tier B mid | 500 | 24,750 | 4,950 | $5 | $3,750 | 0.1% |

All under 0.3% of pool. Completely viable.

### Compute Budget Check

Worst case: Tier A/B, 50 rounds × 5 matches per tx = 250 round evaluations per tx. Each round: strategy lookup, history check, payoff calc, state update (~20 instructions). Total: ~5,000 instructions. Well within Solana's 200K default compute unit limit. No compute budget increase needed.

## 1. Adaptive K Calculation (Contract — `close_registration`)

Replace fixed `matches_per_player` usage with a computed effective K based on participant count.

```rust
fn effective_k(participant_count: u32, config_k: u16) -> u16 {
    if participant_count <= 200 {
        // Tier A: full round-robin
        (participant_count - 1) as u16
    } else {
        // Tier B & C: clamped to [49, 99]
        config_k.min(99).max(49)
    }
}
```

**Where applied:** In `close_registration`, after determining final `participant_count` (post odd-player refund). The effective K is stored in `tournament.matches_per_player` (already snapshotted per tournament). The config value becomes a maximum/hint — the contract overrides it based on actual field size.

**`matches_total` recalculation:** Already computed from `matches_per_player` and `participant_count` via `calculate_match_count()`. No change needed beyond passing the effective K.

## 2. Tiered Round Distribution (Match Logic — `game.rs`)

Replace the fixed 5–15 round range with tier-dependent parameters.

```rust
pub struct RoundConfig {
    pub min_rounds: u8,
    pub max_rounds: u8,
    pub end_probability: u8, // % chance to end each round after min
}

impl RoundConfig {
    /// Tier A & B: longer matches
    pub fn standard() -> Self {
        Self { min_rounds: 20, max_rounds: 50, end_probability: 5 }
    }

    /// Tier C: compressed for scale
    pub fn compressed() -> Self {
        Self { min_rounds: 10, max_rounds: 30, end_probability: 7 }
    }
}
```

**Expected values (geometric distribution):**
- Standard (5% end prob): min 20 + geometric(0.05) capped at 50 → ~35 expected rounds
- Compressed (7% end prob): min 10 + geometric(0.07) capped at 30 → ~20 expected rounds

**Tier selection:** Based on participant count, determined at match execution time. The participant count is available on the Tournament account. Alternatively, store a `round_tier: u8` field on Tournament (set at `close_registration`).

**Option A — derive from participant_count at runtime:**
```rust
let config = if tournament.participant_count <= 1000 {
    RoundConfig::standard()
} else {
    RoundConfig::compressed()
};
```
Pro: no struct change. Con: implicit, threshold buried in match logic.

**Option B — store tier on Tournament:**
```rust
pub round_tier: u8, // 0 = standard (20-50), 1 = compressed (10-30)
```
Pro: explicit, verifiable. Con: struct change → redeploy.

**Recommendation:** Option B. The struct change is trivial (1 byte) and makes the configuration explicit and auditable on-chain.

## 3. Random Sampling Pairing Algorithm (Match Logic — `pairing.rs`)

New pairing method for Tier B (201–1,000 players). Replaces offset method to eliminate structural correlation.

### Algorithm: Greedy Shuffled Sampling

```
generate_random_pairings(n, k, seed):
    // Build all possible pairs
    all_pairs = [(i, j) for 0 ≤ i < j < n]  // n(n-1)/2 pairs
    
    // Deterministic shuffle
    shuffle(all_pairs, seed)
    
    // Greedily select pairs until all players have K matches
    match_count = [0; n]
    selected = []
    
    for (a, b) in all_pairs:
        if match_count[a] < k AND match_count[b] < k:
            selected.push((a, b))
            match_count[a] += 1
            match_count[b] += 1
    
    // Final shuffle for execution order
    shuffle(selected, seed')
    return selected
```

**Properties:**
- Each player gets exactly K matches (assuming K ≤ n−1, guaranteed by tier logic)
- No duplicate pairs
- Opponents uniformly distributed across the field (no adjacency bias)
- Deterministic from seed
- `no_std` compatible (no allocation beyond Vecs)

**Memory concern for Tier C:** At n=5,000, `all_pairs` has 12,497,500 entries. At 8 bytes per pair, that's ~100MB. This is too large for on-chain execution. **Therefore: Tier C retains the existing offset method.** At K=99 with 50 offsets from 2,500 available, the structural correlation is heavily diluted and statistically negligible.

**Memory for Tier B:** At n=1,000, `all_pairs` has 499,500 entries (~4MB). This runs off-chain in the match-logic crate (pairings are generated off-chain by the operator, then verified on-chain via the deterministic seed). 4MB is fine.

**Verification:** Since pairings are deterministic from seed + participant_count + K, anyone can regenerate them off-chain and verify they match the on-chain execution order. No trust assumption changes.

### Pairing Method Selection

```rust
pub fn generate_all_pairings(
    participant_count: u32,
    matches_per_player: u16,
    seed: &[u8; 32],
) -> Vec<(u32, u32)> {
    let n = participant_count as usize;
    let k = matches_per_player as usize;

    if n < 2 {
        return Vec::new();
    }

    if k >= n - 1 {
        // Tier A: full round-robin (with repeated cycles if k > n-1)
        generate_full_round_robin(n, k, seed)
    } else if n <= 1000 {
        // Tier B: random sampling (uniform opponent distribution)
        generate_random_pairings(n, k, seed)
    } else {
        // Tier C: offset method (memory-efficient for large fields)
        generate_offset_pairings(n, k, seed)
    }
}
```

The existing `generate_repeated_round_robin` becomes `generate_full_round_robin` (Tier A). The existing offset logic is preserved as `generate_offset_pairings` (Tier C). `generate_random_pairings` is new (Tier B).

## 4. Full Round-Robin for Tier A (Match Logic — `pairing.rs`)

Simplify the Tier A path. When K = n−1, every player faces every other player exactly once. No repeated cycles needed (unlike the current `generate_repeated_round_robin` which cycles for K > n−1).

```rust
fn generate_full_round_robin(n: usize, k: usize, seed: &[u8; 32]) -> Vec<(u32, u32)> {
    let unique_pairs: usize = n * (n - 1) / 2;
    let matches_per_cycle = n - 1;
    let cycles = (k + matches_per_cycle - 1) / matches_per_cycle;

    let mut base_pairs = Vec::with_capacity(unique_pairs);
    for i in 0..n {
        for j in (i + 1)..n {
            base_pairs.push((i as u32, j as u32));
        }
    }

    let mut pairings = Vec::with_capacity(base_pairs.len() * cycles);
    for _ in 0..cycles {
        pairings.extend_from_slice(&base_pairs);
    }

    let mut rng = SeededRng::new(seed, 0);
    shuffle_pairs(&mut pairings, &mut rng);
    pairings
}
```

This is essentially the existing `generate_repeated_round_robin` renamed for clarity. The key difference from v1.3: K is now set to n−1 by `effective_k()`, so exactly 1 cycle is used for ≤200 players.

## 5. Update `determine_round_count` Signature (Match Logic — `game.rs`)

The round count function needs to accept the round config instead of using hardcoded constants.

```rust
fn determine_round_count(rng: &mut SeededRng, config: &RoundConfig) -> u8 {
    let mut rounds = config.min_rounds;
    while rounds < config.max_rounds {
        if rng.next_percent() < config.end_probability {
            break;
        }
        rounds += 1;
    }
    rounds
}
```

**Callers:** `run_match()` must accept or derive `RoundConfig`. Two options:
- Pass `participant_count` into `run_match()` and derive tier internally
- Pass `RoundConfig` directly

**Recommendation:** Pass `participant_count` — keeps the match-logic crate self-contained. The tier thresholds live in one place.

## 8. Update Operator (Operator)

The operator calls `calculate_match_count()` to determine how many `run_matches` txs to submit. Since `effective_k` changes the match count, the operator must use the same K that the contract computed.

**Current flow:** Operator reads `config.matches_per_player` and calls `calculate_match_count(participant_count, config_k, seed)`.

**New flow:** Operator reads `tournament.matches_per_player` (already snapshotted with the effective K) and uses that. This already works — `close_registration` stores the effective K in the tournament, and `matches_total` is recomputed from it.

**No operator code change needed** beyond recompiling with the updated match-logic crate.

## Risk Assessment

- **Low risk:** All changes are deterministic and verifiable. Pairing algorithms are exercised off-chain first (operator generates, contract replays).
- **Compute budget:** Worst case (50 rounds × 5 matches/tx) uses ~5K of 200K compute units. No risk.
- **Memory (Tier B pairing):** 499,500 pairs at n=1,000 is ~4MB. Generated off-chain by operator — not in the Solana program's heap. The on-chain `run_matches` only calls `get_pairing_for_match(index)` which regenerates pairings per-call. **This is the existing pattern** — `get_pairing_for_match` already calls `generate_all_pairings` and indexes into it. For n=1,000 this may hit BPF memory limits (~32KB heap). **Mitigation:** compute pairing for a given index without generating the full list (see below).
- **BPF heap concern for Tier B:** The greedy shuffled sampling algorithm requires the full pair list in memory. For n=1,000, the all-pairs list alone is 499,500 × 8 bytes = ~4MB — far exceeds BPF's 32KB heap. **Solution:** The pairing for a given match_index must be computable without materializing the full list on-chain. Two approaches:
  - **A) Precompute and store pairings off-chain, pass as instruction data.** Operator includes `(player_a_index, player_b_index)` in the `run_matches` instruction. Contract verifies the pair is valid by re-deriving it. But re-deriving requires the full list — circular.
  - **B) Use a streaming/indexable algorithm** that can produce the K-th selected pair without storing all pairs. This is non-trivial for the greedy sampling approach.
  - **C) Use the offset method for all on-chain tiers** and implement random sampling as an off-chain verification guarantee only. Since pairings are deterministic from seed, anyone can verify off-chain that the offset method was applied correctly. The statistical improvement of random sampling over offset at K=99 is marginal.

**Revised recommendation:** Use the **offset method for all tiers** on-chain. The random sampling algorithm is elegant but impractical within BPF heap constraints. At K=99 with 50 offsets out of n/2 available, the adjacency correlation is diluted to statistical insignificance for n > 200. Reserve true random sampling for a future version with off-chain proof verification (e.g., ZK proofs or a verifier program).

### Updated Tier Summary (Post Risk Assessment)

| Tier | Player Count | K | Rounds | Pairing |
|------|-------------|---|--------|---------|
| **A** | n ≤ 200 | n − 1 | 20–50 | Full round-robin |
| **B** | 201–1,000 | 99 | 20–50 | Offset (50 of n/2 offsets) |
| **C** | 1,001–5,000 | 99 | 10–30 | Offset (50 of 2,500 offsets) |

This simplifies implementation: items 3 and 4 from the implementation order are reduced to refactoring the existing offset code rather than writing a new algorithm. The offset method already works correctly — the key improvements are adaptive K and longer matches.

## What This Achieves

| Metric | v1.3 (current) | v1.4 (proposed) |
|--------|----------------|-----------------|
| Opponent coverage (n=100) | 16% (16 of 99) | **100%** (full RR) |
| Opponent coverage (n=500) | 16% | **20%** (99 of 499) |
| Opponent coverage (n=5000) | 16% | **2%** (99 of 4999) |
| Rounds/match (expected) | ~10 | **~20–35** |
| Total rounds/player (n=100) | ~160 | **~3,465** |
| Total rounds/player (n=5000) | ~160 | **~1,980** |
| Pairing bias | Structural (offset adjacency) | Diluted (many offsets) / eliminated (full RR) |
| Operator cost % of pool | <0.01% | **<0.3%** |

The biggest win is Tier A: tournaments under 200 players get Axelrod-quality full round-robin with ~35 rounds per match. This is where most early tournaments will land.
